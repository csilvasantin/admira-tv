Para integradores

El contrato: un evento normalizado

Todo el valor del portal viaja en un único evento. El mismo que hoy emite la demo desde el navegador, mañana lo emitirá el router server-side. Uno solo, idempotente.

POST /gamification/api/eventsapplication/json
// Check-in de presencia emitido por el portal cautivo
{
  "eventId":    "wifi-<visitorId>-<locationId>-<YYYYMMDD>",
  "type":       "wifi.checkin",
  "subjectId":  "<visitorId anónimo>",
  "sourceApp":  "socialwifi-web",
  "metadata": {
    "locationId": "demo-cafe-central",
    "screenId":   "scr-demo-01"   // opcional
  },
  "occurredAt": "2026-07-09T09:41:00Z"
}

// Respuesta del puente admira-loyalty
{ "ok": true, "duplicate": false, "status": "pending" }

Idempotente por día

La ventana canónica del check-in es 1 vez al día por espacio. El eventId es determinista (visitante + espacio + fecha): reenviarlo el mismo día devuelve duplicate:true y no acuña puntos otra vez. Los +10 pts se acuñan en ≤5 min por cron; la respuesta inmediata es status:"pending".

Nota de arquitectura

El proxy es same-origin (/gamification/api/events) y hace de puente al backend real de fidelización. La demo emite el evento desde el navegador; en producto, el portal cautivo del hardware de red enviará este mismo evento server-side. Portal en producto: en integración con el hardware de red.
